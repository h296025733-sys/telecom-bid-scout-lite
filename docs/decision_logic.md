# 判断逻辑说明

本文档只描述当前代码已经实现的判断流程，主要对应：

- `src/scorer.py`
- `config/rules.yaml`
- `src/reporter.py`
- `src/state.py`
- `data/sample_bids.csv`

## 1. 项目判断流程图

```text
CSV 候选公告
  -> 读取 title / summary / 日期 / 来源 / 链接 / 官方入口
  -> 加载 YAML 规则
  -> 匹配通信关键词与高价值关键词
  -> 匹配排除关键词
  -> 检查发布时间与截止时间
  -> 检查来源是否命中可信来源
  -> 检查 url 与 official_entry
  -> 检查 bid id 是否已在 state.json 中记录
  -> 计算分数并生成 reasons / matched_keywords
  -> 应用阈值与风险约束
  -> 输出 值得跟进 / 需人工复核 / 不建议跟进
  -> 生成 outputs/report.md
  -> 可选推送未重复的待关注线索到飞书
  -> 更新 outputs/state.json
```

## 2. 评分规则说明

当前规则由 `config/rules.yaml` 配置，评分由 `src/scorer.py` 执行。关键词匹配范围是 `title` 与 `summary` 拼接后的文本；可信来源匹配范围是 `source`。

| 判断项 | 当前代码行为 |
| --- | --- |
| 命中 `include_keywords` | 加 `min(30, 12 + 命中数量 * 6)` 分 |
| 未命中 `include_keywords` | 扣 18 分 |
| 命中 `high_value_keywords` | 加 `min(24, 命中数量 * 8)` 分 |
| 命中 `exclude_keywords` | 扣 35 分 |
| 发布时间可解析且不晚于当前日期 | 加 6 分 |
| 发布时间缺失、不可解析或晚于当前日期 | 扣 10 分 |
| 截止时间可解析且未过期 | 加 20 分 |
| 截止时间已过期 | 扣 35 分 |
| 截止时间缺失或不可解析 | 扣 14 分 |
| 来源命中 `trusted_sources` | 加 16 分 |
| 来源未命中可信来源 | 扣 8 分 |
| 存在 `official_entry` | 加 18 分 |
| 无官方入口但存在 `url` | 加 6 分 |
| `url` 与 `official_entry` 都没有 | 扣 18 分 |

最终分数会被限制在 0 到 100 之间。当前配置阈值是：

- `min_score_follow: 70`
- `min_score_review: 35`

`state.json` 中的重复状态不改变分数。重复线索会在 `reasons` 中留下说明，并在主流程的飞书推送队列中被过滤掉。

## 3. 三类判断结果说明

### 值得跟进

当前实现中，分数达到“值得跟进”阈值，并且有明确 `official_entry` 的线索，才会保留为“值得跟进”。

典型特征是：

- 标题或摘要明确命中通信业务方向；
- 截止时间仍有效；
- 来源可信；
- 有官方公告或报名入口。

### 需人工复核

当前实现中，分数达到人工复核阈值但未达到跟进阈值，会被归为“需人工复核”。

另一个重要场景是：某条线索本来分数已经达到“值得跟进”，但没有 `official_entry`，代码会把它降级为“需人工复核”。这类线索可能业务方向正确，但还需要人工确认原始公告、报名入口或主体限制。

### 不建议跟进

当前实现中，低于人工复核阈值的线索会被归为“不建议跟进”。

另外有两个风险约束会直接影响结果：

- 截止时间已过期时，结果会被设为“不建议跟进”；
- 命中排除关键词且没有任何通信关键词支撑时，结果会被设为“不建议跟进”。

## 4. 样例线索判断过程

### 样例 A：呼叫中心与外呼系统公告

`BID-2026-001` 的标题和摘要命中：

- `呼叫中心`
- `外呼系统`

这两个词同时属于通信方向关键词和高价值关键词。样例数据中还提供了：

- 有效的发布时间；
- 未过期的截止时间；
- `公共资源交易中心` 可信来源；
- 官方入口。

因此它在样例报告中被判断为“值得跟进”。

### 样例 B：转载的线路资源征集信息

`BID-2026-005` 命中：

- `通信服务`
- `线路资源`

但它的来源是 `行业资讯转载站`，并且 `official_entry` 为空。代码会给它保留候选链接的分数，但不会把它当作已经确认的官方入口，因此更适合进入“需人工复核”。

### 样例 C：打印机、物业和食堂采购

`BID-2026-006` 未命中通信方向关键词，却命中：

- `打印机`
- `物业`
- `保洁`
- `食堂`

这类信息即使来自可信采购来源，也会因为业务方向不相关而被判断为“不建议跟进”。

### 样例 D：过期语音线路公告

`BID-2025-004` 命中语音线路和通信服务相关词，也有可信来源与官方入口。但样例截止时间是 `2025-03-20`，对当前演示日期已过期，因此最终结果为“不建议跟进”。

## 5. 为什么它不是空壳项目

这个项目已经有一条实际可运行的处理链路：

- 能读取 CSV 候选公告；
- 能从 YAML 加载可维护规则；
- 能对每条线索输出分数、分级、命中词和原因；
- 能生成可阅读的 Markdown 报告；
- 能用 `state.json` 记录处理过的 bid id；
- 能在推送前过滤重复待关注线索；
- 能在配置飞书 Webhook 后发出通知，未配置时也能正常跳过。

它的重点不是做全量招采平台，而是先把“候选数据进来以后如何稳定筛选和输出”做成最小闭环。

## 6. 和 OpenClaw 的分工关系

OpenClaw 适合做开放式上游工作：

- 定时搜索关键词；
- 读取网页；
- 整理候选公告字段；
- 把候选结果写入 CSV；
- 调用项目命令。

本项目适合做稳定规则执行层：

- 读取候选数据；
- 运行固定评分规则；
- 输出可解释判断；
- 生成报告；
- 管理推送去重；
- 执行可选飞书通知。

这种拆分把搜索探索和规则输出分开，便于迭代上游数据来源，同时保留下游判断的可复核性。

## 7. 面试时可以如何解释

可以这样说明：

> 我做的不是一个大而全的招采平台，而是通信行业招采线索筛选的轻量闭环。上游可以由 OpenClaw 或搜索流程提供候选公告，下游由 Python 工具读取 CSV 和 YAML 规则，结合通信关键词、排除词、日期有效性、来源可信度和官方入口做评分与分级。每条结果都保留判断原因和命中关键词，再输出 Markdown 报告，并通过状态文件避免重复推送。这样既能展示业务规则拆解，也能展示自动化流程真正跑通。
